# Jietu Rules

本文件是 `jietu` 仓库的规范文档，面向后续维护和改动约束。它不是面向最终使用者的快速启动说明；`AGENTS.md` 应把代理先引导到这里，`README.md` 继续承担使用入口，而这里定义的是实现边界、兼容要求、回归风险和验证标准。

## 1. 项目定位

- 这是一个本地网页截图归档工具，当前核心目标仍然是围绕 `shokz.com` 做按时间留档、回看和变更识别。
- 归档主流程不能被“变更检测”取代。截图归档和变更检测是主从关系，不是二选一关系。
- 默认工作模式是高回归风险项目：优先最小改动，禁止无关重构，任何结构升级都要先考虑旧数据和旧调用兼容。

## 2. 默认协作规则

以下要求默认长期生效，除非需求里明确覆盖。`AGENTS.md` 可以保留精简版入口说明，但不应和本文件产生独立分叉：

1. 不要直接改代码，先阅读相关实现并做影响分析。
2. 先说明需要改哪些文件、为什么改、可能影响哪些旧功能。
3. 优先采用最小改动方案，禁止无关重构。
4. 不要修改无关文件。
5. 如果改动会超出当前需求范围，先停下来说明原因。
6. 改完后必须做构建、测试、lint 或类型检查，按项目实际情况选择。
7. 最后必须说明改了哪些文件、验证结果、剩余风险。
8. 如果用户明确给出“保持不变的功能”或“允许修改的路径”，必须严格遵守。

## 3. 当前已验证的稳定契约

### 3.1 运行入口

- `npm start`：启动本地服务。
- `npm run capture`：执行一次手动截图。
- `npm run compare`：基于已有截图重建 `data/changes.json`。
- `npm run repair:data`：修复运行时派生展示字段，并立即重建变更数据。
- `npm test`：当前仓库的测试主入口。

### 3.2 已公开接口

- `GET /api/state`：返回当前配置、抓图状态、截图列表和 `changesSummary`。
- `GET /api/changes`：返回完整变更记录。
- `POST /api/capture`：默认只读，只有设置 `PAGE_SHOT_ADMIN=1` 后才允许写操作。
- `POST /api/config`：默认只读，只有设置 `PAGE_SHOT_ADMIN=1` 后才允许写操作。
- `DELETE /api/snapshots/:id`：允许删除单次截图，并触发变更汇总重算。

### 3.3 数据与输出位置

- `archive/`：原始截图与派生 diff 图的归档根目录。
- `data/snapshots.json`：截图索引，属于归档主流程的事实来源。
- `data/changes.json`：变更检测输出，属于附加分析结果，不是截图归档的替代品。
- `logs/`：运行日志目录。

## 4. 已确认的不变业务约束

### 4.1 归档主流程必须保持不变

- 新截图继续写入 `archive/`。
- `data/snapshots.json` 继续作为截图归档索引。
- 删除、预览、回看历史截图的能力必须保留。

### 4.2 变更检测是增量模块

- `data/changes.json`、`GET /api/changes`、`changesSummary` 继续承担“对归档结果做附加对比”的职责。
- 不允许为了新功能把对比系统改造成新的主存储或替代归档流程。
- diff 图应继续输出到 `archive/diffs/YYYY-MM-DD/<site>/...png` 一类的派生路径，不能覆盖原截图。

### 4.3 比对规则不能倒退

- 基线规则默认是“上一张同位截图”，不是固定基线图。
- 变更判断默认依赖“文本 + 视觉”联合信号，不引入 OCR 作为默认链路。
- 同位判断依赖稳定逻辑位置信息，特别是 `sectionKey`、`tabIndex`、分页/hover 等状态字段。
- 文本比较依赖 `textBlocks` 等元数据，不能退回到只看像素 diff。
- 非 banner 的默认 related state 不能被粗暴丢弃，否则会破坏重命名类变更的前后配对。

## 5. 新规范

本节定义后续改动必须对齐的目标规范。若当前工作树中存在进行中的实现，可将其视为“正在向本规范收敛”，但不能把所有目标项都当作已经稳定发布的事实。

### 5.1 配置结构以 v2 为唯一内部规范

- `targets[]`：只描述抓图目标，如 `id`、`url`、`label` 以及可选的 `captureMode`、`fullPage` 覆盖。
- `deviceProfiles[]`：只描述项目内启用的设备，如 `id`、`platform`、`devicePresetId`、`enabled`。
- `capturePlans[]`：显式描述 `target x deviceProfile` 的执行矩阵，如 `id`、`targetId`、`deviceProfileId`、`enabled` 以及可选覆盖项。
- `intervalMinutes`、`waitAfterLoadMs`、`dismissPopups` 等共享抓图参数仍保留为顶层配置。

### 5.2 必须兼容旧配置

- 旧版 `urls + devicePresetId + viewport` 结构仍要能被读取。
- 兼容策略优先采用“运行时归一化到 v2”，而不是强迫用户手工改写旧 `data/config.json`。
- 只有真正发生保存动作时，才应把配置落盘为 v2，避免无意义改写 tracked 配置以外的数据。

### 5.3 平台分治只拆必要层

- 后续实现以 `pc` / `mobile` 为主轴做执行分治，但只拆必要边界。
- 抓图服务层保留调度和公共入口职责，不应继续堆叠大量 PC/mobile 条件分支。
- 底层浏览器控制、PNG、diff 等能力应尽量复用，而不是为分平台重复造轮子。

### 5.4 运行时元数据必须可回传

- 新截图记录需要带上 `platform`、`deviceProfileId`、`capturePlanId` 这类显式元数据。
- 变更记录里的 location、from、to 也应尽量透传相同维度，方便 API 和前端按平台聚合。
- 历史 snapshot 或 change 缺字段时，允许用旧逻辑兜底推导，但不要强制回写全部旧数据。

### 5.5 API 与前端对齐方向

- `/api/state` 需要能够承载 v2 `config`，并为前端提供按平台聚合的派生视图。
- `/api/changes` 保持直接读取变更结果的职责，不和状态接口混写成单一大接口。
- 前端可以继续保持单页，但平台视图应按 `PC / 手机` 明确分治，不再长期依赖单一设备筛选模型。

## 6. 典型高回归风险点

- `shokz` 首页 related sections 的 mobile-only 逻辑。
- 导航相关截图的 desktop-only 逻辑。
- 同位匹配中依赖 `tabIndex`、hover 状态、分页位置的配对规则。
- 历史数据缺少 `platform`、`deviceProfileId`、`capturePlanId` 时的回退显示和修复逻辑。
- 手动删除快照后，`data/changes.json` 和界面摘要是否仍能一致重建。

## 7. 变更边界

- 默认只改与当前需求直接相关的文件。
- 文档改动不应顺手调整代码结构。
- 代码改动若会波及归档语义、变更对比契约、旧配置兼容或前端聚合模型，必须先做影响分析。
- 若需求会扩大到“重写 compare 算法”“重构整套 capture pipeline”“替换前端信息架构”，应先停下来说明，不直接扩写范围。

## 8. 验证要求

每次功能改动或缺陷修复后，至少按改动类型执行相应验证；文档改动也应做最小必要校验。

- JavaScript 改动：`node --check` 覆盖被修改文件。
- 逻辑改动：`npm test`。
- 结构性改动：`git diff --check`，必要时补一个短时本地 API smoke check。
- 涉及 `/api/state` 或 `/api/changes`：至少验证接口可返回 200，并确认返回结构未破坏现有消费方。

## 9. Git 与交付规则

- 仓库要求在完成功能改动或 bug 修复后提交到 `main`，并在提交成功后推送 `origin/main`。
- `archive/`、`logs/`、`data/snapshots.json` 属于运行时生成数据，应继续保持忽略，不要纳入版本控制。
- 如果交付阶段再次遇到 `.git/index.lock` 或其他权限类阻塞，应明确报告外部阻塞，不要回退已经验证通过的工作树改动。

## 10. 文档关系

- `AGENTS.md`：代理入口文件，应保持精简，并明确先读 `rules.md`。
- `README.md`：面向使用者的启动、命令和接口说明。
- `rules.md`：面向维护者和代理的唯一详细规范文档。
- 如果 `README.md` 和 `rules.md` 存在冲突，维护约束以 `rules.md` 为准。
- `AGENTS.md` 不应复制出一套和 `rules.md` 相互漂移的长规范；它应做入口和摘要，而不是替代本文件。

## 11. 截图预览交付契约

- 每个正式截图的点开预览，在页面存在有意义的隐藏状态时，必须优先生成 composite 总览图。隐藏状态包括轮播图、Tab、产品媒体缩略图、抽屉、hover 状态、颜色/型号切换等。
- composite 总览图由左右两个视觉部分组成：左边是一列完整长截图，右边是同一页面对应模块的“更多截图”状态序列。
- 右侧状态截图必须和左侧长截图里的对应模块保持纵向高度对齐。代码层面以来源模块的 `clip.y` / `sourceClip.y` 作为对齐契约。
- 如果页面没有有意义的轮播、Tab 或更多状态，不要强行生成假的右栏；预览只展示单列长截图。
- 前端卡片在存在 `snapshot.homeOverview` 时必须优先使用它作为主预览。单独的 `relatedShots` 只是生成 composite 的来源或兜底细节，不应替代 composite 成为主要展示效果。
- 如果某个快速模式主动跳过 related-state 截图，那么它不能被声明为满足“左长图 + 右侧更多截图对齐”的交付契约。
